## 1. السؤالان اللذان يُربكان كل وسيط


**السؤال الأول** (يسأله الوسيط):
> "علي ومحمد يستخدمان قيد. أنا وسيط بينهما. لديّ رصيد شخصي مع علي من تعاملات سابقة. هل التحويل سيُغيّر هذا الرصيد؟"

**السؤال الثاني** (يسأله المُرسِل أو المُستقبِل):
> "علي طلب من محمد (الوسيط) أن يُرسل مالاً لبديع — كلهم يستخدمون قيد. أين سيظهر هذا التحويل؟ هل في محادثة علي مع محمد؟ أم في محادثة علي مع بديع مباشرة؟"

الإجابتان مترابطتان وكلتاهما تكشف عن تصميم هندسي عميق.

## 2. محادثة علي مع بديع — لا محادثة علي مع محمد

الكود في `ListAccountStatementChatUseCase` يُطبّق **قاعدة الاستبعاد** التالية:

```dart
// Rule 3: Exclude any voucher where the Mediator is the Counterparty
if (v.isTripartite) {
  final mediatorId = v.tripartiteMeta?.mediatorAccountId ?? v.affectedAccountId;
  if (myId == mediatorId || cpId == mediatorId) {
    return false; // Exclude from Mediator's chat with parties
  }
}
```

**ماذا تعني هذه القاعدة؟**

السند الثلاثي **لا يظهر** في محادثة الوسيط مع أي من الطرفَيْن. بدلاً من ذلك، يظهر **في محادثة علي مع بديع مباشرة**، مع إشارة إلى أن محمد هو وسيط هذه المعاملة.

**الصورة الكاملة لما يرى كل طرف**:

```
محادثة علي ↔ بديع:
  ✓ يظهر التحويل ← "أرسل علي 1,000 ريال عبر وسيط محمد"
  علي: outgoing ← أنا المصدر
  بديع: incoming ← أنا الوجهة

محادثة علي ↔ محمد:
  ✗ لا يظهر التحويل الثلاثي هنا
  (يظهر فقط إذا كان بينهما تعاملات شخصية مباشرة أخرى)

محادثة محمد ↔ بديع:
  ✗ لا يظهر التحويل الثلاثي هنا أيضاً
```

## 3. كيف "يقرأ" الكود اتجاه المعاملة؟

`_directionFromPerspective` يحل إشكالية التوجيه عبر `linkedPartyId`:

```dart
if (v.isTripartite) {
  final isReceipt = v.type == VoucherType.receipt;
  // سند القبض: counterpartyId=A، linkedPartyId=B
  final sourceId = isReceipt ? v.counterpartyId : v.tripartiteMeta!.linkedPartyId;
  // سند الدفع: counterpartyId=B، linkedPartyId=A
  final destId = isReceipt ? v.tripartiteMeta!.linkedPartyId : v.counterpartyId;

  if (perspectiveId == sourceId) return 'outgoing'; // يرى نفسه مُرسِلاً
  if (perspectiveId == destId) return 'incoming';   // يرى نفسه مُستقبِلاً
}
```

بغض النظر عن أي السندَيْن (القبض أو الدفع) يُعرض، `linkedPartyId` يُتيح للنظام تحديد "من هو المصدر الحقيقي" و"من هو الوجهة الحقيقية" — فيُقدّم التحويل في سياقه الصحيح.

## 4. ما الذي يظهر في كل محادثة؟ — الصورة الكاملة

**علي يفتح محادثته مع بديع**:
```
بديع
────────────────────────────────
[تحويل ثلاثي] 1,000 ريال
↑ صادر منك لبديع
عبر الوسيط: محمد
────────────────────────────────
```

**بديع يفتح محادثته مع علي**:
```
علي
────────────────────────────────
[تحويل ثلاثي] 1,000 ريال
↓ واردٌ إليك من علي
عبر الوسيط: محمد
────────────────────────────────
```

**محمد يفتح محادثته مع علي**: التحويل الثلاثي **لا يظهر هنا**. تظهر فقط تعاملاتهما الشخصية المباشرة.

**محمد يفتح محادثته مع بديع**: التحويل الثلاثي **لا يظهر هنا أيضاً**.

---

## 5. محمد يرى التحويل — لكن في مكان مختلف

محمد يرى التحويل في **صفحة التحويلات الثلاثية** أو في **كشف صندوقه النقدي**، لا في محادثاته الشخصية مع الطرفَيْن. هناك يرى:

| السند | الطرف | الاتجاه | المبلغ |
|------|-------|---------|--------|
| القبض | علي | وارد لصندوقه | +1,000 |
| الدفع | بديع | صادر من صندوقه | -1,000 |

**رصيده مع علي الشخصي**: لم يتغير.
**رصيده مع بديع الشخصي**: لم يتغير.

---

## 6. سيناريو التقديم — حين يتغير كل شيء

إذا اختار محمد "تقديم المبلغ" بدلاً من الوساطة الحقيقية — أي يدفع من ماله لبديع قبل استلام المال من علي:

هذه **ليست وساطة** — هذا إقراض. والقيود ستكون مختلفة جوهرياً:

- **سند الدفع**: محمد → بديع (سند عادي، **لا ثلاثي**).
- **ذمة لمحمد على علي**: قيد يُسجّل أن علياً يدين لمحمد بالمبلغ.

**ما الذي سيظهر في المحادثات؟**:
- محادثة محمد ↔ بديع: يظهر سند الدفع هنا.
- محادثة محمد ↔ علي: هنا تظهر الذمة — "علي يديني 1,000 ريال".
- محادثة علي ↔ بديع: **لا يظهر فيها شيء** — محمد هنا ليس وسيطاً بل مُقرِضاً.

هذا هو الفارق الجوهري بين الحالتَيْن من منظور العلاقات الموثَّقة.

## 7. جدول المقارنة الكامل — الوساطة مقابل التقديم

| المعيار | الوساطة الحقيقية | التقديم |
|---------|----------------|---------|
| محادثة علي ↔ بديع | ✓ التحويل يظهر مباشرة | ✗ لا يظهر شيء |
| محادثة علي ↔ محمد | ✗ التحويل لا يظهر | ✓ الذمة تظهر هنا |
| محادثة محمد ↔ بديع | ✗ التحويل لا يظهر | ✓ سند الدفع يظهر هنا |
| رصيد محمد مع علي | ✗ لم يتغير | ✓ تغيّر — أصبح علي يديني |
| رصيد محمد مع بديع | ✗ لم يتغير | ✓ تغيّر — دفعتُ لبديع |
| القفل الشرطي | ✓ مُفعَّل | ✗ غير موجود |

## 8. الخلاصة — أربعة مبادئ مُطبَّقة في الكود

**مبدأ ١ — الظهور المباشر**: التحويل الثلاثي يظهر في محادثة المُرسِل مع المُستقبِل مباشرة — لا في محادثاتهما مع الوسيط. هذا مُطبَّق عبر قاعدة الاستبعاد في `ListAccountStatementChatUseCase`.

**مبدأ ٢ — محمد في الهامش**: دور الوسيط يظهر كـ"معلومة إضافية" على التحويل (`mediatorName`) — لا كطرف رئيسي في المحادثة. هو القناة، لا الطرف.

**مبدأ ٣ — العزل المحاسبي**: حسابات محمد الشخصية مع علي أو بديع لا تتأثر بالوساطة أبداً. القيود تمر عبر صندوقه النقدي حصراً.

**مبدأ ٤ — التمييز من الاختيار**: الفرق بين الوساطة والتقديم ليس تقنياً فحسب — إنه يُغيّر أين يظهر التوثيق، ومن يرتبط بمن، وكيف تتشكّل العلاقات الموثَّقة في النظام.
